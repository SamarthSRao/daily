# OpenTrace — Project Specification
## Distributed Tracing System (Jaeger-Equivalent)

**Role:** Final Year Project + Portfolio Centrepiece  
**Stack:** Go · TypeScript · gRPC · ClickHouse · Kafka · Next.js · Kubernetes

---

## 1. Project Overview

OpenTrace is a complete open-source distributed tracing system built in Go and TypeScript — the portfolio centrepiece and final year project. It is architecturally equivalent to Jaeger: it receives OTLP spans over gRPC and HTTP, processes them through a Kafka pipeline with tail-based sampling, stores them in ClickHouse (10M spans/sec, p99 query < 200ms on 30-day window), and serves them via a gRPC query service to a Next.js UI with trace waterfalls, service maps, and live tail. OpenTrace instruments itself — the self-referential demo is live.

## 2. Key Demonstrations

- OTLP-compatible — any OpenTelemetry SDK sends spans with zero config changes
- 10M spans/sec ingestion throughput (ClickHouse bulk insert, verified with k6)
- Tail-based sampling — keeps 100% errors + slow spans, samples 5% of normal traces
- ClickHouse columnar storage with monthly partitioning, partition pruning verified
- Self-instrumentation — OpenTrace traces its own 7 components live
- K8s Operator — all 7 components managed by a single Kubernetes Operator

## 4. Technical Depth

### OTLP Collector — gRPC + HTTP Receiver

The Collector implements the OpenTelemetry Protocol (OTLP) `TraceService` gRPC service. It receives `ExportTraceServiceRequest`, validates spans (required fields, valid timestamps, trace ID format), and publishes to the `spans.raw` Kafka topic. Partition key = `trace_id` ensures all spans of a trace go to the same partition — ordering is guaranteed without sorting. Backpressure: if Kafka is slow, the Collector returns `RESOURCE_EXHAUSTED`, documented in the ADR.

### Tail-Based Sampling

Tail-based sampling makes a sampling decision after the full trace arrives, not on the first span. The Pipeline Processor buffers spans per `trace_id` for 10 seconds. At decision time:
- Keep **100%** of traces with any error span
- Keep **100%** of traces with any span > 500ms
- Sample **5%** of remaining normal traces

This ensures all errors and slow traces are captured regardless of volume, while reducing storage cost for healthy traffic.

### ClickHouse Storage — Columnar at Scale

The spans table uses the MergeTree engine with `PARTITION BY toYYYYMM(start_time)`. Partition pruning is verified with `EXPLAIN`: a 24-hour query reads 1–2 partitions out of 7 total. Bulk insert batches 10,000 spans before writing — single-row vs batch benchmark shows 100x throughput difference, documented in `BENCHMARKS.md`. TTL policy: spans older than 30 days are automatically deleted. Storage cost vs Elasticsearch: **8x cheaper per GB**.

```sql
CREATE TABLE spans (
    trace_id        FixedString(32),
    span_id         FixedString(16),
    parent_span_id  FixedString(16),
    operation_name  LowCardinality(String),
    service_name    LowCardinality(String),
    start_time      DateTime64(6),
    duration_us     UInt64,
    status_code     UInt8,
    attributes      Map(String, String),
    resource        Map(String, String)
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(start_time)
ORDER BY (service_name, start_time, trace_id)
TTL start_time + INTERVAL 30 DAY;
```

### Self-Instrumentation Demo

OpenTrace instruments its own 7 components using the Auto-Instrumentation SDK. A live trace of a query request flows through: UI → API Gateway → Query Service → ClickHouse — visible inside OpenTrace itself. The service map shows the internal dependency graph. The live tail streams spans in real time as they are generated by a demo microservice. This is the most compelling demo artifact: the system traces itself.

## 5. Data Model

- **ClickHouse spans table:** `trace_id, span_id, parent_span_id, service_name, operation_name, start_time, duration_us, status_code, tags Map(String,String), logs Array(String)`
- `PARTITION BY toYYYYMM(start_time)` — monthly partitions, TTL 30 days
- **PostgreSQL span_metadata:** `trace_id, service_name, ingested_at` — range partitioned by month
- **Redis trace cache:** `trace:{traceId}` → serialised spans, TTL 5 min, TTL jitter ±30s
- **Kafka topics:** `spans.raw` (partition key: trace_id), `spans.sampled`, `spans.errors`

## 7. Non-Negotiable Rules

- `go test -race ./...` — data races in the Collector lose spans silently under load
- `goleak.VerifyNone(t)` — goroutine leaks in the span pipeline accumulate and crash production
- Outbox pattern for every Kafka publish — crash between receive and publish loses spans forever
- `EXPLAIN` on every ClickHouse query — partition pruning verified (Selected 1–2/7 parts)
- 10M spans/sec load test documented in `BENCHMARKS.md` — untested claims are fiction
- OTel compatibility verified — official OTel Go SDK sends to OpenTrace with zero config changes
